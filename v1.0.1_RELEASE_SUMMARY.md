### âœ… 2. "When NOT to Use" Section (README) - CRITICAL

**Added:** Explicit limitations section with honest explanations

**Content:**

#### âŒ Multi-Node / Clustered Deployments
- **Problem:** In-memory storage means limits aren't shared across servers
- **Example:** 3 servers Ã— 60 req/min = 180 req/min actual limit
- **Solution:** Use Redis-based solution (coming in v2.0)

#### âŒ Very High Throughput
- **Safe:** Up to ~5,000 req/sec
- **Caution:** 5,000-10,000 req/sec (monitor memory)
- **Not recommended:** >10,000 req/sec
- **Reason:** Timestamp tracking memory overhead

#### âŒ Not a Replacement for API Gateways
- **Problem:** Application-level, not network-level
- **Impact:** Malicious requests still reach your Node.js process
- **Solution:** Use ratewarden WITH an API gateway, not instead of

#### âŒ Exact Fairness Guarantees
- **Problem:** Sliding window is approximate (Â±1-2%)
- **When it matters:** Billing tied to exact counts
